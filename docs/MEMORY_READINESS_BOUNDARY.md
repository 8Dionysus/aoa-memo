# Memory Readiness Boundary

## Purpose

This note defines the memo-side holding boundary for future high-pressure
memory questions.

It prepares `aoa-memo` to answer what existing memory objects can hold and what
must route to neighboring owner repositories, without starting any future
protocol, planting a seed pack, changing schemas, or creating a live ledger.

This is not a live memory ledger.
It is not a protocol.
It is not proof.

memory is not proof.

## Readiness judgment

`aoa-memo` is ready as a doctrine and surface layer for the first memory
readiness questions:

- which existing memory objects should hold future reviewed consequences,
  deltas, retention checks, and recall anchors
- which neighboring repo owns each stronger judgment
- which pressures need proof or runtime support before they become live

`aoa-memo` is not ready, and should not pretend to be ready, as a live state
store, verdict layer, retention worker, role-rights layer, or graph substrate.

That split is the current safe posture.

## Owner Split

- `Agents-of-Abyss` owns center-level protocol posture and ecosystem-level
  meaning.
- `aoa-agents` owns role contracts, actor rights, office authority, and any
  later civil/service or agonic actor variants.
- `aoa-playbooks` owns scenario choreography and campaign-shaped routes.
- `abyss-stack` owns live stores, runtime retention jobs, backups, and local
  execution plumbing.
- `aoa-evals` owns proof surfaces, scoring, verdict meaning, and promotion
  evidence.
- `aoa-kag` owns downstream derived substrate lifts and graph-facing
  normalization.
- `aoa-routing` owns navigation hints and return hops.
- `aoa-memo` owns memory objects, provenance, lifecycle posture, temporal
  relevance, recall contracts, and memo-side writeback doctrine.

## Memory Pressure Map

Use the existing canon before inventing new object families.

| future pressure | memo-side landing | guardrail |
| --- | --- | --- |
| session or run memory | `episode`, with `state_capsule` only for exported working state | do not store live scratchpad residue |
| governed gate, closure, or transition | `decision` | do not turn memo wording into verdict logic |
| durable reviewed consequence, including a later scar term if a stronger owner introduces it | `audit_event` for lifecycle change, `pattern` or `claim` only after review, and `provenance_thread` for walkback | do not create a `scar` object kind now |
| memory delta | `inquiry_checkpoint.memory_delta_refs`, `state_capsule`, `episode`, or `decision` | keep memory deltas distinct from canon deltas |
| canon delta reference | `inquiry_checkpoint.canon_delta_refs` only by reference | do not make memo a ToS canon author |
| retention check | `audit_event` or `episode` plus lifecycle, `current_recall`, and temperature posture | runtime retention policy and workers stay in `abyss-stack` |
| unresolved contradiction | lifecycle posture, contradiction refs, audit walkback, and provenance walkback | preserve tension; do not narrate false resolution |
| survivor or bridge candidate | `bridge` only when stronger source refs and review posture are explicit | downstream KAG lift stays in `aoa-kag` |
| civil/service assistant trace | `episode`, `decision`, or `audit_event` when a governed service action matters beyond one run | agent identity and service contract meaning stay in `aoa-agents` |

## What May Happen Now

Now, without starting any future protocol, `aoa-memo` may:

- route humans and agents to the existing writeback and recurrence surfaces
- keep future durable-consequence, delta, retention, and recall pressure mapped
  to current memory objects
- preserve memory versus canon delta separation
- keep provenance and lifecycle posture visible
- hand recall and contradiction guardrails to `aoa-evals` for bounded proof
- hand bridge candidates to `aoa-kag` only as source-owned memo exports
- let `aoa-routing` point back to memo recall contracts without copying memo
  meaning

## What Must Wait

These moves should wait for owner-layer evidence:

- a dedicated `scar` object family
- a live memory ledger
- verdict language inside `aoa-memo`
- automatic promotion from memory to canon
- runtime retention jobs inside `aoa-memo`
- graph normalization of scar or survivor memory inside `aoa-memo`
- role-rights policy inside `aoa-memo`

## Proof Expectations

The existing downstream pilots in `aoa-evals` are the right first proof path:

- `aoa-memo-recall-integrity`
- `aoa-memo-contradiction-integrity`

They cover recall precision, provenance visibility, staleness honesty, and
contradiction-visible consumption.

They do not yet prove a full scar or retention program.
If a later protocol requires stronger scar or retention guarantees,
`aoa-evals` should own that proof surface and `aoa-memo` should provide only
the bounded memory cases, examples, or contracts needed by that proof.

## Practical Rule

When high-pressure memory need appears, ask:

1. Is this an event, a claim, a decision, a pattern, a bridge, an audit event,
   or a provenance thread?
2. What stronger source or reviewed trace explains it?
3. Is it a memory delta, a canon delta reference, or both kept separate?
4. Does recall need current, historical, or withdrawn posture?
5. Does proof, routing, graph lift, role authority, or runtime retention belong
   to a neighboring repo instead?

If the answer requires a new object family, live ledger, verdict, or automatic
promotion rule, stop at a reviewed owner-layer proposal.
